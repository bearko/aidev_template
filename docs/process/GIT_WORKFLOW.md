# Git ワークフロー（ブランチ・コミット・PR・マージ）

特定のホスティングサービスやブランチ戦略ツールに依存しない **汎用ルール** です。チーム名物に合わせて調整してよい。

## 1. ブランチ戦略（推奨の考え方）

### 長命ブランチ

- **`main`（または `trunk`）**: 常に **デプロイ可能またはリリース可能** な状態を目標にする。直接コミットは禁止または例外のみとする。
- **リリースブランチ**（任意）:  semver や日付で切る。パッチのみ入れるなど、プロジェクトのリリース方針に従う。

### 短命ブランチ（作業用）

- 1 つの論理的変更（1 SPEC または関連する小さなセット）につき 1 ブランチを推奨。
- 命名例（プレフィックスは統一する）:
  - `feature/<short-topic>` — 新機能・仕様実装
  - `fix/<short-topic>` — 不具合修正
  - `docs/<short-topic>` — 文書のみ
  - `chore/<short-topic>` — ツール設定など、挙動に影響しない変更

## 2. コミット

### メッセージ規約（Conventional Commits 互換の例）

```
<type>(<scope>): <短い説明>

[本文: なぜ変更したか、SPEC 参照、破壊的変更の注意]

```

- **type の例**: `feat`, `fix`, `docs`, `test`, `chore`, `refactor`, `perf`, `ci`
- **scope**: 任意。モジュール名や SPEC ID（例: `spec-007`）でもよい。
- **言語**: チームの合意言語で統一（英語推奨なら全体で英語）。

### コミット粒度

- レビューしやすい単位に分割する。機械的な整形と意味のある変更を同一コミットに混ぜない。

## 3. Pull Request（マージリクエスト）

- **ドラフト**: 作業中はドラフトまたは「WIP」ラベルで開き、レビュー可能になったら Ready にする。
- **本文に含めるもの**:
  - 対象 SPEC（リンクまたは ID）
  - 変更概要
  - 受け入れ基準への対応（チェックリスト）
  - テスト・確認の証跡（[TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)）
  - 破壊的変更・移行手順（該当時）
- **サイズ**: 大きい PR は分割を検討。分割できない場合は本文で「レビュー順序」を示す。

## 4. レビューとマージ

### マージ前の人間レビュー（デフォルト）

- **`main`（および保護された対象ブランチ）へマージする前に、人間によるコードレビュー（または同等の監査可能な承認）を必須とする。** CI が緑であることは必要条件にすぎず、**CI の成功だけをもってマージしてよいとはしない**（静的解析やテストは人間が見ない変更を検知しきれないため）。
- **自動マージ（auto-merge）** を使う場合も、**人間の承認（approve）を条件に含める**こと。承認なしでマージが完了する設定は、意図的な例外として手順書に理由と範囲を書いたうえでのみ許容する。
- **エージェント・ボットがマージボタンを押す**運用にする場合は、**その前に必ず人間が同一 PR をレビュー済み**であること。エージェントのみの承認で本番ラインへ入れることはデフォルトでは認めない。
- 上記を **ブランチ保護ルール**（必須レビュー数、ステータスチェック、管理者によるバイパス禁止など）で機械的に強制できるなら、テンプレ採用プロジェクトはそれを推奨する。ルールが無い場合でも、チームは本節を **プロセス上の必須** と読み替えて運用する。

### レビュー人数とマージ方法

- **必須レビュー人数**: プロジェクトで定める（**デフォルトの目安: 1 人以上の人間**）。自己承認のみを認めるかはポリシーで明示する。
- **マージ戦略**:
  - **Squash**: 履歴を線形に保ちたい場合に有効。コミットメッセージは PR タイトルに合わせて整理。
  - **Merge commit**: ブランチの経緯を残したい場合。
  - **Rebase merge**: 線形履歴かつ個別コミットを残す場合。
- チームは **1 つのデフォルト** を選び、例外を README か本書に記載する。

## 5. マージ前チェックリスト（自動 + 手動）

- 自動: ビルド、静的解析、テスト（存在する範囲で）が緑であること。
- **人間レビュー**: 少なくとも 1 名のレビュアーが PR を確認し、**承認（approve）またはチームが定める同等の記録**が付いていること（上記セクション「マージ前の人間レビュー」）。
- 手動: SPEC・憲章との整合、[HUMAN_IN_THE_LOOP.md](./HUMAN_IN_THE_LOOP.md) で止めるべき局面がクリアであること。

## 6. リバートとホットフィックス

- **リバート**: 問題のコミット / PR を明示し、新規 PR で戻す。事後分析をチケットまたは文書に残す（任意だが推奨）。
- **ホットフィックス**: `main` から短命ブランチを切り、最小差分で修正。通常開発ブランチへバックポートする方針を決めておく。

## 7. タグとリリース

- リリースタグ（例: `v1.2.0`）はリリース担当ロールが付与。タグに紐づく成果物（バイナリ、SBOM 等）はプロジェクト方針に従う。
