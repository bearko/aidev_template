# エージェント運用ガイド（AI / 自動化）

人間とエージェントが **同じ憲章と仕様** の下で協働するための境界と習慣です。

## 1. 入力として必須に近いもの

- 対象 **SPEC ID** または文書へのリンク。
- **意図**: 何を達成し、何を変えないか（非目標）。
- **制約**: 期限、触ってはいけない領域、環境（本番禁止など）。

## 2. 自律的に進めてよい範囲（広め）

本テンプレートは「PR を作るところまでは自律」を原則とする：

- SPEC と憲章に沿った実装・リファクタ・テスト・型・リント
- ドキュメントの誤字補正、明らかな抜けの追加
- 依存関係の minor / patch 更新（major は PR 本文で論点化）
- Jira / GitHub Issues の起票・状態遷移・コメント
- ブランチ作成、コミット、`develop` 等保護ブランチへの push（保護ルールが PR 経由を強制する）
- PR の作成・編集・コメント

## 3. 止まるべき範囲（最小化）

技術的に不可避な箇所のみ：

- 秘密情報の生成・取得・コミット（鍵・パスワード・生トークン）
- 破壊的シェル操作（`rm -rf`, `git push --force` to protected branches, `git reset --hard`, `gh repo delete`）
- 外部 API への書き込み（決済・メール送信・本番 DB 直接操作）

それ以外は **PR を作って人間レビューに委ねる**。「重要そうだから止まる」ではなく「重要そうだから PR 本文に明記する」。

## 4. PR で論点を提示する作法

[HUMAN_IN_THE_LOOP.md](./HUMAN_IN_THE_LOOP.md) の論点に該当する変更を含む場合：

1. **作業は完走させる**（テスト追加・リント修正まで）
2. PR 本文の「マージ前確認事項（HITL）」セクションに該当項目をチェック
3. 該当項目の **理由・代替案・推奨選択** を箇条書きで併記
4. 人間レビュアに「マージ可否」を委ねる

## 5. 出力の品質

- 変更理由を PR 本文またはコミット本文に残す（人間が書くのと同じ基準）
- 不確実な点は「仮定」として列挙し、検証方法を添える
- スクリーンショット・CI リンク・ローカル実行ログなどの証跡を貼る

## 6. 仕様の更新

- SPEC の **承認済み** 本文をエージェントが直接変える場合は、**SPEC 改訂 PR を併出** する（実装 PR と分離）
- ポリシーがない場合のデフォルト: エージェントは **草案 PR** で SPEC 修正案を出し、人間がマージする

## 7. トレーサビリティ

- 自動生成コードであっても、レビューとテスト証跡は人間の変更と同等に扱う（[TESTING_STRATEGY.md](../testing/TESTING_STRATEGY.md)）
- コミット末尾に `Co-Authored-By: <model-name>` を含めて来歴を明示する慣習を推奨

## 8. ツール側の補助

`.claude/settings.json`（[テンプレート](../../.claude/settings.json)）で本ガイドを permission 設定として実装する：

- `allow`: 自律範囲（コード編集、git/gh 操作、Jira 書き込み、CI 操作）
- `deny`: 止まるべき範囲（破壊的操作、秘密情報）
- `ask`: PR マージ系のみ（`gh pr merge`, `gh pr review --approve`）
